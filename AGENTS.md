# follow-up 项目规则

## 强制执行前置检查
在开始规划或执行任何任务之前，**必须优先检查 `.agents/skills` 目录**中是否存在可用 skill。绝对禁止跳过此步骤直接执行任务或编写新代码。

## Conda 环境
本项目的 conda 环境为 `follow-up`，启动命令为 `conda activate follow-up`

## 语言强制规则
1. 你必须始终使用中文（简体中文）进行所有回复
2. 即使用户使用英文或其他语言提问，你也必须用中文回答
3. 技术术语、代码变量名、API 名称可保留英文，但必须在首次出现时提供中文解释
4. 禁止在回复中混合使用其他语言进行解释（除非用户明确要求翻译）

## 训练可视化与日志 (强制标准)

 **训练脚本日志与监控规范**

为保证长时间实验在前台与后台（如 nohup）场景下均可被实时监控、及时排错并完整追溯，所有训练脚本必须严格遵守以下规范。

 **1\. 实时可见性要求**

所有训练过程必须同时满足以下两点：

* **终端实时输出**：关键训练信息必须实时打印到终端。  
* **日志文件实时写入**：相同信息必须同步写入 logs/{script\_name}.log，并保证可通过 tail \-f 实时查看。

**具体要求如下：**

* 终端输出必须使用 print(..., flush=True)。  
* 日志文件写入必须使用行缓冲模式，例如 open(log\_path, "w", buffering=1)。  
* 每次写入日志后必须立即 flush()。  
* 建议所有训练脚本统一通过 python \-u 启动，以进一步避免缓冲导致的输出延迟。

 **2\. 进度显示要求**

训练脚本应提供清晰的训练进度显示。

在交互式终端中，推荐使用：

tqdm(..., desc="Training", ncols=80, file=sys.stderr)

*  
* **无论是否启用 tqdm，都必须保留纯文本形式的 epoch 日志输出**，确保日志文件内容清晰可读。

 **3\. 日志输出频率**

训练日志必须满足以下最低频率要求：

* 每个 epoch 至少输出一次基础训练状态。  
* 每 10 个 epoch 至少输出一次完整核心指标。  
* 若总训练轮数较少（如 epoch \<= 50），建议每个 epoch 都输出完整核心指标。

 **4\. 每个 epoch 必须输出的基础信息**

每个 epoch 的日志至少应包含以下字段：

* Epoch 编号  
* Seed  
* 当前学习率（Learning Rate）  
* Train Loss  
* Val Loss（若有验证集）  
* 当前最佳验证指标（如 Best Val AUPRC / AUROC）  
* 单个 epoch 耗时

**推荐输出格式如下：**

Epoch 12/100 | Seed 42 | LR 1.0e-3 | TrainLoss 0.1834 | ValLoss 0.2017 | BestValAUPRC 0.0450 | Time 8.2s

 **5\. 核心指标输出要求**

除基础 loss 外，训练过程中还必须定期输出当前任务的核心指标。对于本项目，建议至少包括：

* AUPRC  
* AUROC  
* F1（若当前阶段已定义阈值）  
* Precision / Recall（如适用）  
* 其他当前阶段关注的主指标

*注：若当前阶段尚不适合输出完整指标，也必须明确说明原因，不能仅输出 loss 而完全缺少性能反馈。*

 **6\. 后台运行规范**

所有正式实验建议通过如下方式启动：

nohup python \-u run\_xxx.py \> logs/run\_xxx.log 2\>&1 &

**要求：**

* 日志路径必须清晰、可区分不同实验。  
* 文件名建议包含：脚本名、模型名、seed、时间戳或 run\_id。

**例如：**

logs/train\_causal\_tabdiff\_seed42\_20260311\_1530.log

 **7\. 不推荐的实现方式**

为避免输出延迟、日志混乱或训练过程不可见，当前项目阶段**不建议**使用 Python logging 模块作为主日志方案，除非已经明确验证其实时性和稳定性。

**以下做法一律禁止：**

* 不带 flush=True 的关键训练打印  
* 写入日志后不立即 flush()  
* 训练结束后一次性补写日志  
* 仅保存最终结果，不记录训练过程  
* 仅使用进度条而不保留纯文本 epoch 日志

 **8\. 验收标准**

一个训练脚本只有在满足以下条件时，才可视为符合规范：

* \[ \] 能在终端实时看到训练进展  
* \[ \] 能通过 tail \-f logs/xxx.log 实时看到日志更新  
* \[ \] 每个 epoch 至少有基础状态输出  
* \[ \] 每隔固定轮数有核心指标输出  
* \[ \] 日志中能够追溯：  
  * 当前实验配置  
  * 当前 seed  
  * 当前训练进度  
  * 当前最佳性能  
  * 是否出现异常
